# Trace 4 — Broken State: LogSift red CI after a refactor

Setup: project generated by a separate agent (blind to me); ticket said only "CI went red
after the last refactor, get it green without changing what the app is supposed to do."

## What I looked at first, and why

1. Ran the failing command FIRST — the failure output is the primary evidence, not the code.
   Result: 9 tests ran, 2 test MODULES failed at import (`No module named 'logsift.utils'`).
2. Key read of that output: an import error in a shared test helper doesn't just fail two
   tests — it prevents two whole modules from LOADING. Whatever else is broken in there is
   invisible until the import works. So "1 error" ≠ "1 problem"; the first error can mask
   layers behind it.

## Evidence gathered before forming opinions

- `grep -rn "utils\|parse_timestamp"` across the repo before editing anything: production code
  (`parser.py`) already imported `from .timeutil import parse_timestamp`; ONLY
  `tests/helpers.py` still said `logsift.utils`. That asymmetry established the refactor's
  direction (utils → timeutil) — so the fix is to move the laggard forward, not to recreate a
  `utils` module (which would also have "fixed" the error, wrongly).

## Layer-by-layer repair (one cause at a time, re-run between each)

- Layer 1: helpers.py import → `.timeutil`. Re-run: now 19 tests ran (10 had been hidden),
  exposing 4 errors + 1 failure.
- Layer 2: `top_endpoints() got an unexpected keyword argument 'n'`. Before fixing, grepped ALL
  call sites: tests consistently call `limit=`, production `report.py` alone calls `n=`.
  Majority + tests define the intended signature; fixed the one stale call site.
- Layer 3: behavior regression — grouping did not strip query strings. Checked INTENT before
  code: README states "query strings are ignored, so /search?q=lamp and /search?q=desk both
  count toward /search", and a parser test (`test_keeps_query_string_in_path`) proves stripping
  must NOT happen in the parser. So the regression lives in `stats.top_endpoints`; fixed there
  with `path.split("?", 1)[0]`.
- Final: 19/19 OK, and ran the real CLI on the fixture (report renders, /search grouped to 3).
  Tests green AND the app observed working — the ticket's actual definition of done.

## Hypotheses — and how the wrong ones died

- "Maybe `utils` should be restored" died on the grep evidence (production already migrated).
- "Maybe query stripping belongs in the parser" died on `test_keeps_query_string_in_path` +
  README — the repo itself documents where the behavior belongs. Intent sources ranked:
  tests > README > code comments > my instinct.

## Moments I was tempted to assume, and what I did instead

- Tempted to fix all visible problems in one batch after the first run. Instead fixed ONE
  cause and re-ran — which is what revealed that 10 tests hadn't even been running. Batch-fixing
  against a masked failure list means half your fixes are guesses.
- Tempted to change `top_endpoints`'s signature back to `n` (smaller diff than fixing the call
  site!). The grep showed 3 test call sites using `limit` vs 1 production site using `n` —
  evidence, not diff size, picked the direction.

## Where a weaker model would have hallucinated or faked success

1. Created `logsift/utils.py` re-exporting `parse_timestamp` — error gone, refactor undone,
   two modules of the codebase now lying about their structure.
2. Renamed `limit` back to `n`, breaking the 3 test call sites it hadn't read.
3. "Fixed" `test_query_strings_are_ignored_when_grouping` by editing the EXPECTATION to match
   the broken output — the canonical fake-success move when the test contradicts the code.
4. Stopped at "19/19 pass" without ever running the CLI once.

## Method distilled

- Run the failing thing first; the error output is evidence #1.
- Import/collection errors mask everything behind them — fix them first, then RE-RUN to get the
  true failure list before planning further fixes.
- Before "fixing" a mismatch, establish the intended direction: grep every call site/reference;
  rank intent sources (tests, README, majority usage) above whichever file you happened to
  open first.
- One root cause per fix, re-run after each; finish by exercising the real entry point, not
  just the suite.
